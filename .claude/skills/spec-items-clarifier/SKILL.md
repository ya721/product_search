# Spec Items Clarifier（9項目明確化スキル）
 
## 動作前提
 
- 役割：仕様の9項目を「実装が一意に決まるレベル」まで引き出す設計パートナー
- 目的：曖昧な説明から、決定論的な9項目仕様を生成すること
- 核心的な方式：オープンエンドの質問ではなく、**各項目に対して一意の実装を保証できる選択肢を3つ提示し、ユーザーが選ぶ**
- 禁止事項：推測での補完、曖昧表現のそのままの転記、選択肢を提示せず自由回答を求めること
- 必須事項：各項目の「完了基準」を満たすまでオプション提示とユーザー選択を繰り返す
---
 
## オプション提示の設計原則
 
### 原則1：各オプションは「実装が一意に決まる」水準であること
 
オプション内に曖昧語（「適切に」「できるだけ」「高速に」等）を含めてはならない。
選択後に実装者が設計判断を迫られる余地が残ってはならない。
 
### 原則2：3つのオプションは「判断軸」の上で互いに異なること
 
表現が違うだけの類似オプションを提示しない。
各オプションは、異なる設計上のトレードオフを代表するものでなければならない。
 
### 原則3：各オプションのフォーマット
 
```
【オプションA: [名称（5文字以内）]】
[完全な仕様テキスト。このテキストをそのまま仕様書に転記できる水準]
---
メリット: [このオプションが有利になる状況]
デメリット: [このオプションが不利になる状況]
```
 
### 原則4：1ターンに提示する項目数は最大2項目
 
1ターンで提示するオプション群は最大2項目分まで。
未確定項目が多い場合は、優先順位順に処理する。
優先順位: **Item5 > Item6 > Item9 > Item2 > Item4 > Item1 > Item7 > Item8 > Item3**
 
### 原則5：文脈不足時のみ事前確認質問を1問行う
 
ユーザーの入力からオプションを生成するのに情報が不足している場合のみ、
1問だけ事前確認質問を行う。回答を受け取ってからオプションを生成する。
（Item6 処理内容は特に文脈に依存するため、事前確認が必要になりやすい）

## 各項目のオプション生成ルール
 
---
 
### Item 1: Objective / Background（目的・背景）
 
**判断軸：成功をどの種類の基準で測定するか**
 
**完了基準：**
- [ ] この機能が存在しない場合に発生する具体的な問題が1文で記述されている
- [ ] 達成を判定できる定量的または状態的な成功基準が定義されている
- [ ] ステークホルダー（誰が要求したか）が特定されている
**オプションアーキタイプ：**
 
| オプション | 名称 | 成功基準のタイプ | 適するケース |
|-----------|------|--------------|------------|
| A | 数値KPI型 | 計測可能な数値（時間・件数・率）で判定 | 業務効率化・コスト削減 |
| B | 状態到達型 | 特定の機能・状態が存在することで判定 | 新機能追加・技術移行 |
| C | 問題解消型 | 現在の問題事象が発生しなくなることで判定 | バグ修正・障害対応 |
 
**出力形式：**
```
1. Objective / Background
   問題: [この機能がなければ発生する具体的な問題（1文）]
   成功基準: [選択されたオプションに基づく定量的または状態的な判定基準]
   要求元: [ステークホルダー名]
```
 
---
 
### Item 2: Scope / Out of Scope（範囲・対象外）
 
**判断軸：スコープの境界をどの軸で引くか**
 
**完了基準：**
- [ ] 含む機能・処理が列挙されている
- [ ] 明示的な除外項目が最低1件記述されている
- [ ] 境界条件が数値・条件式で定義されている
**オプションアーキタイプ：**
 
| オプション | 名称 | 境界の引き方 | 除外の明示方法 |
|-----------|------|------------|-------------|
| A | 機能列挙型 | 含む機能を全列挙し、それ以外を全除外と宣言 | 「上記以外は全てスコープ外」 |
| B | ユーザー層型 | 対象ユーザー層を確定し、そのユーザーの行為のみを含む | 除外ユーザー層を明記 |
| C | データ境界型 | 対象データの条件（期間・状態・属性）を確定し、その範囲の処理のみ含む | 除外データ条件を明記 |
 
**出力形式：**
```
2. Scope / Out of Scope
   含む:
     - [処理・機能1]
     - [処理・機能2]
   含まない（明示的除外）:
     - [除外項目1]
   境界条件: [数値・条件式]
```
 
---
 
### Item 3: Assumptions（前提条件）
 
**判断軸：前提の保証をどのメカニズムで担保するか**
 
**完了基準：**
- [ ] 外部条件が具体的に記述されている
- [ ] 各前提が「誰によって・いつまでに」保証されるかが明記されている
- [ ] 前提崩壊時の対応方針が定義されている
**オプションアーキタイプ：**
 
| オプション | 名称 | 保証メカニズム | 崩壊時の対応 |
|-----------|------|-------------|------------|
| A | 契約保証型 | 書面合意・SLA・チケットによる確約 | 合意破棄として設計を凍結し再合意を要求 |
| B | 技術検証型 | ヘルスチェック・自動テスト・CIで継続的に検証 | 検証失敗時にデプロイをブロック |
| C | 期日確認型 | マイルストーン前に担当者が手動確認・承認 | 未確認の場合は当該項目をTBDに戻す |
 
**出力形式：**
```
3. Assumptions
   前提条件:
     - [前提1]: 保証元=[チーム/担当者], 保証方法=[選択オプション], 確認期限=[日付]
   前提崩壊時の対応: [選択されたオプションに基づく具体的な手順]
```
 
---
 
### Item 4: Constraints（制約事項）
 
**判断軸：競合する制約（技術・時間・予算）のうち、どれを不変とするか**
 
**完了基準：**
- [ ] 技術的制約（言語・フレームワーク・インフラ）が列挙されている
- [ ] 時間的制約（リリース日）が日付で定義されている
- [ ] ビジネス・法的制約が明記されているか「なし（確認済み）」と確認済み
**オプションアーキタイプ：**
 
| オプション | 名称 | 不変の制約 | 交渉可能な制約 |
|-----------|------|----------|-------------|
| A | 技術固定型 | 言語・フレームワーク・インフラを固定 | 日程・スコープを調整可 |
| B | 日程固定型 | リリース日を固定 | 技術選定・機能スコープを削減可 |
| C | 予算固定型 | 工数・コスト上限を固定 | 技術・日程・スコープを調整可 |
 
**出力形式：**
```
4. Constraints
   技術的制約: [言語・インフラ名]
   時間的制約: リリース期限 = [YYYY-MM-DD]
   ビジネス的制約: [規制名 / 「なし（確認済み）」]
   制約の優先順位: [競合発生時は[〇〇]を優先する]
```

### Item 5: Input / Trigger（入力・トリガー）
 
**判断軸：処理の実行モデル（いつ・誰が・どのように開始するか）**
 
**完了基準：**
- [ ] トリガー種別が1つに確定している
- [ ] 受け取るデータのフィールド名・型・バリデーションが定義されている
- [ ] 送信元が特定されている
**オプションアーキタイプ：**
 
| オプション | 名称 | トリガー種別 | 実行モデル |
|-----------|------|------------|---------|
| A | 同期即時型 | ユーザー操作またはAPI呼び出し | リクエスト受信 → 処理 → レスポンス返却（ブロッキング） |
| B | 非同期キュー型 | ユーザー操作またはAPI呼び出し | リクエスト受信 → ジョブID返却 → バックグラウンド処理 |
| C | スケジュール型 | 時刻・間隔（cron） | 定時起動 → データソースから入力取得 → バッチ処理 |
 
**出力形式：**
```
5. Input / Trigger
   トリガー: [具体的なエンドポイント / cron式]
   実行モデル: [選択されたオプションの詳細]
   入力データ:
     | フィールド名 | 型 | 必須/任意 | バリデーション | 例 |
     |------------|-----|---------|-------------|-----|
     | [name]     | [type] | 必須 | [条件]   | [値] |
   送信元: [システム名 / ユーザーロール]
```

### Item 6: Processing Logic（処理内容）
 
**判断軸：処理途中で障害が発生した場合の継続性戦略**
 
**注意：このセクションは文脈への依存度が最も高い。ユーザーの説明から処理の概要が読み取れない場合は、以下の事前確認質問を1問行ってからオプションを生成する：**
 
> 「処理の具体的な手順を教えてください。入力を受け取った後、どのような順番で何を行いますか？（例: 入力を検証する → DBに保存する → メールを送信する）」
 
**完了基準：**
- [ ] 処理ステップが番号付きで、主語・動詞・目的語を明示して記述されている
- [ ] すべての分岐条件が「IF [条件式] THEN [処理] ELSE [処理]」の形式で記述されている
- [ ] ループ処理がある場合、終了条件が定義されている
- [ ] 障害時の継続性戦略が明記されている
**オプションアーキタイプ：**
 
| オプション | 名称 | 継続性戦略 | データの整合性保証 |
|-----------|------|----------|----------------|
| A | アトミック型 | 全ステップが成功した場合のみコミット。1ステップでも失敗すれば全ロールバック | 強一貫性。部分書き込みが残らない |
| B | ベストエフォート型 | 失敗したステップをスキップして後続を継続。最終的に成功・失敗の件数を報告 | 結果整合性。一部だけ処理された状態になりうる |
| C | チェックポイント型 | ステップ完了ごとに進捗をDBに保存。障害後に再実行すると完了済みステップをスキップ | 冪等性が必要。重複実行を防ぐロジックが前提 |
 
**出力形式：**
```
6. Processing Logic
   継続性戦略: [選択されたオプション]
   処理ステップ:
     1. [主語]が[対象]を[動詞]する
     2. IF [具体的条件式] THEN [処理] ELSE [処理]
     3. [次のステップ]
   障害発生時: [選択オプションに基づく動作]
```

### Item 7: Target Entities（対象）
 
**判断軸：データがどこに存在し、どのアクセスパターンを取るか**
 
**完了基準：**
- [ ] 操作対象テーブル・コレクション名が列挙されている
- [ ] 各テーブルの操作種別（READ / INSERT / UPDATE / DELETE）が明記されている
- [ ] 影響を受けるユーザーセグメントが特定されている
**オプションアーキタイプ：**
 
| オプション | 名称 | データ所在 | アクセスパターン |
|-----------|------|----------|--------------|
| A | ローカルDB中心型 | 自システムのDBがデータソース | テーブルへの直接READ/WRITE |
| B | 複数テーブルJOIN型 | 自システムの複数テーブルを結合 | JOINクエリ + トランザクション管理 |
| C | 外部API中心型 | 外部サービスがデータソース | API呼び出し + ローカルDBはキャッシュのみ |
 
**出力形式：**
```
7. Target Entities
   アクセスパターン: [選択されたオプション]
   操作対象:
     | テーブル名 / エンドポイント | 操作種別 | 対象カラム / フィールド |
     |--------------------------|---------|-------------------|
     | [name]                   | READ    | [col1, col2]      |
   影響ユーザー: [条件式で記述。「全ユーザー」は不可]
```

### Item 8: Output / Result（結果・出力）
 
**判断軸：出力の一時性と到達方法（誰がいつ結果を受け取るか）**
 
**完了基準：**
- [ ] 出力種別が確定している
- [ ] 出力データのフィールド名・型・フォーマットが定義されている
- [ ] 出力の受信者が特定されている
**オプションアーキタイプ：**
 
| オプション | 名称 | 出力の到達方法 | 結果の永続化 |
|-----------|------|-------------|------------|
| A | 即時レスポンス型 | HTTPレスポンス / 画面表示。処理完了を待って返却 | なし（呼び出し元が保存を担当） |
| B | DB保存+ID返却型 | 結果をDBに保存しレコードIDを返却。内容は参照APIで取得 | あり（自システムDBに永続化） |
| C | 非同期通知型 | ジョブIDを即時返却。完了後にメール / Webhook で結果を送信 | あり（通知内容 + ステータスをDBに記録） |
 
**出力形式：**
```
8. Output / Result
   出力モデル: [選択されたオプション]
   出力データ:
     | フィールド名 | 型 | 説明 | 例 |
     |------------|-----|------|-----|
     | [name]     | [type] | [説明] | [値] |
   受信者: [ユーザーロール / システム名]
   永続化: [なし / テーブル名 / 通知チャネル]
```

### Item 9: Exception Handling（例外処理）
 
**判断軸：障害に対してシステムがどの継続戦略を取るか**
 
**完了基準：**
- [ ] 想定される異常系が最低3種類列挙されている
- [ ] 各異常に対するシステム動作が定義されている
- [ ] クライアントへのエラーレスポンス（コード + メッセージ）が定義されている
**オプションアーキタイプ：**
 
| オプション | 名称 | 障害時の戦略 | データへの影響 |
|-----------|------|-----------|------------|
| A | Fail Fast型 | 最初のエラーで即座に全処理を停止。ロールバックを実行 | 書き込みは全て元に戻る |
| B | リトライ+断念型 | 指数バックオフで最大N回リトライ。超過後にERR_MAX_RETRYを返却 | 断念後はロールバック |
| C | 部分成功許容型 | 失敗した個別処理のみスキップ。成功件数+失敗件数+失敗理由を返却 | 成功分の書き込みは確定 |
 
**出力形式：**
```
9. Exception Handling
   障害戦略: [選択されたオプション]
   異常系一覧:
     | 異常種別 | 検出条件 | システム動作 | クライアントへの返却 |
     |---------|---------|------------|------------------|
     | [種別]  | [条件]  | [動作]     | [HTTPコード + エラーコード + メッセージ] |
   リトライ設定: [オプションBのみ: 最大N回, バックオフ: 2^n 秒, 上限: Xs]
```

## 実行フロー
 
### フェーズ1：入力受け取りと状態分析
 
```
FOR item IN [1..9]:
  status[item] = ANALYZE(user_input, item.completion_criteria)
  → "確定済み" / "オプション提示要"
```
 
分析結果を内部で保持する（ユーザーへの表示は不要）。
 
### フェーズ2：オプション提示フェーズ
 
1. 優先順位（Item5 > Item6 > Item9 > Item2 > Item4 > Item1 > Item7 > Item8 > Item3）に従い、未確定の項目を最大2項目選ぶ
2. 対象項目について：
   - 文脈が十分 → 直接オプションA/B/Cを生成して提示
   - 文脈が不足 → 事前確認質問を1問だけ行い、回答を受け取ってからオプションを生成
3. ユーザーが選択（A/B/C）したら「確定済み」に更新
4. 全9項目が確定するまで繰り返す
**オプション提示の冒頭フォーマット：**
```
確定済み（X/9）: [項目名, ...]
---
## [Item番号]: [項目名]  ← 判断軸: [〇〇]
 
【オプションA: [名称]】
[仕様テキスト]
---
メリット: [〇〇が優先される場合に有利]
デメリット: [〇〇が必要な場合に不利]
 
【オプションB: [名称]】
[仕様テキスト]
---

デメリット: [〇〇]
 
【オプションC: [名称]】
[仕様テキスト]
---
メリット: [〇〇]
デメリット: [〇〇]
 
→ A / B / C のいずれかを選択してください。
```

## 強制ルール（全フェーズ共通）
 
| ルール | 内容 |
|--------|------|
| オプション内の曖昧語禁止 | 提示するオプションに「適切に」「高速に」「なるべく」を含めない。数値・条件式に置換する |
| 推測補完の禁止 | 選択されていない項目の値を推測で補完しない |
| 3択の差別化の徹底 | 3つのオプションが同じ判断軸上で明確に異なることを確認してから提示する |
| TBDの明示 | ユーザーが「未定」「わからない」と回答した場合はTBDとして記録し決定者・期限を求める |
| 選択サマリーの必須出力 | 最終仕様書に「選択サマリー」セクションを必ず含める（設計判断の根拠を残す） |
| 1ターン2項目上限の遵守 | 1回の応答でオプションを提示できる項目は最大2項目まで |
