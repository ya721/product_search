## チェック項目

### 1. Model層の検証
- [ ] **データ管理の責務**: Modelはデータの保持、取得、永続化のみを担当しているか
- [ ] **UI依存性の排除**: ModelはViewやViewModelに関する知識を持っていないか
- [ ] **ビジネスロジックの配置**: ドメインロジックがModel層に適切に配置されているか
- [ ] **データ通知機能**: データ変更時の通知機能（Observable、NotifyPropertyChanged等）が実装されているか
- [ ] **単体テスト可能性**: ModelクラスがUI依存なしで単独テスト可能か

### 2. View層の検証
- [ ] **UI表現の専念**: ViewはUI要素の表示と基本的なユーザー操作の受付のみを担当しているか
- [ ] **ロジック排除**: Viewにビジネスロジックやデータ変換処理が含まれていないか
- [ ] **データバインディング**: ViewModelとの連携がデータバインディングで実現されているか
- [ ] **直接参照の回避**: ViewがModelを直接参照していないか
- [ ] **イベント処理**: ユーザーイベントがCommandパターンでViewModelに委譲されているか

### 3. ViewModel層の検証
- [ ] **仲介役の実装**: ViewとModelの橋渡し機能が適切に実装されているか
- [ ] **プレゼンテーションロジック**: UI表示用のデータ変換・フォーマット処理が含まれているか
- [ ] **Command実装**: ユーザー操作に対応するCommandが実装されているか
- [ ] **プロパティ通知**: Viewへのデータ変更通知（INotifyPropertyChanged等）が実装されているか
- [ ] **View非依存**: ViewModelがView固有のUI要素に依存していないか
- [ ] **状態管理**: View表示に必要な状態の管理が適切に行われているか

### 4. 依存関係の検証
- [ ] **単方向依存**: View → ViewModel → Model の単方向依存が守られているか
- [ ] **循環参照回避**: 各層間で循環参照が発生していないか
- [ ] **疎結合の実現**: インターフェースや依存性注入による疎結合が実現されているか
- [ ] **テスト容易性**: 各層が独立してテスト可能な構造になっているか

### 5. アンチパターンの検出
- [ ] **Fat ViewModel**: ViewModelが肥大化していないか（300行以上は要注意）
- [ ] **God Object**: 単一クラスに過度な責務が集中していないか
- [ ] **Anemic Model**: Modelがデータの器だけになっていないか
- [ ] **Logic in View**: ViewにIF文やループ処理が大量に含まれていないか
- [ ] **Direct DB Access**: ViewModelがデータベースに直接アクセスしていないか

## 評価方法

各項目について以下の基準で評価してください：

- ✅ **準拠**: MVVMパターンに完全に従っている
- ⚠️ **要改善**: 軽微な違反があるが修正可能
- ❌ **違反**: 重大な設計違反がある

## 注意事項

- 単純なGetterやSetterの指摘よりも、アーキテクチャの本質的な問題に注目する
- フレームワーク固有の制約がある場合は考慮する
- パフォーマンスとアーキテクチャのバランスを考慮した現実的な提案を行う
- 段階的な改善計画を提示する
