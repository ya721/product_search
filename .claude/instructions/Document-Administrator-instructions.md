## 核心原則

### 1. 条件付き有効性の認識

- ドキュメントは万能薬でも無用の長物でもない
- ドキュメントの価値は文脈、複雑さ依存する
- 投資対効果を常に考慮し、機械的な適用を避ける

### 2. 実証的証拠に基づく判断

- 開発者は構造、可読性、ドキュメントを重要視するが、経験レベルで認識が異なる
- 複雑なコードではドキュメントが有効、単純なコードでは限定的
- 保守段階ではコードコメントが最も有用な成果物

### 3. アジャイル原則との調和

- 「動作するソフトウェア > 包括的ドキュメント」を尊重
- ただし「ドキュメント不要」ではなく「適切なドキュメント」を目指す
- 継続的改善と文脈適応を重視

### 4. 必須ドキュメント化規則

- 関数・構造体・プロトコル・クラスには例外なくドキュメントコメントを追加
- 統一された形式で一貫性を保持
- 理解に必要な全要素を網羅的に記述

### 5. ドキュメント必須項目

- 役割と責務: 何のために存在するか
- 関数の目的: 何を実現するか
- パラメータの型と意味: 入力の詳細仕様
- 戻り値: 出力の型と意味
- 例外: 発生する可能性のあるエラー
- 使用例: 実際の呼び出し方法

### 6. 品質基準

- 完全性: 必須項目の全てを記述
- 正確性: 実装との完全な整合性
- 明確性: 曖昧さを排除した記述

## 文脈評価フレームワーク

### プロジェクト特性の評価
```
複雑度: [低/中/高] - アルゴリズム、業務ロジックの複雑さ
規模: [小/中/大] - コードベース規模
期間: [短期/中期/長期] - プロジェクト継続期間
ドメイン: [汎用/専門] - 業界特有の知識の必要性
```

### ドキュメント優先度マトリックス
- **高優先度**: 複雑度高 + 規模大 + 長期 + 離職率高 + 専門ドメイン
- **中優先度**: 上記の一部条件が当てはまる場合
- **低優先度**: 複雑度低 + 規模小 + 短期 + 離職率低 + 汎用ドメイン

## ドキュメント記述ガイドライン

### A. 必須ドキュメント（高優先度時）

#### 1. API/関数ドキュメント
```
/**
 * [機能の要約を1行で]
 * 
 * [詳細説明 - WHYとHOWの両方]
 * 
 * @param {type} paramName - パラメータの説明と制約
 * @returns {type} 戻り値の説明と可能な値
 * @throws {ErrorType} 例外条件と対処法
 * @example
 * // 具体的な使用例
 * const result = functionName(example);
 */
```

#### 2. 複雑なアルゴリズムのコメント
```
// WHYの説明: なぜこのアプローチを選択したか
// トレードオフ: 他の選択肢とその理由
// 参考資料: アルゴリズムの出典やさらなる情報源

// 実装の各ステップ
// ステップ1: 初期化の理由
// ステップ2: ループ条件の意図
// ステップ3: 終了条件の重要性
```

#### 3. 設計決定記録（ADR形式）
```
## 設計決定: [決定内容]
- **状況**: 直面していた問題
- **決定**: 選択した解決策
- **理由**: なぜこの選択をしたか
- **結果**: 予想される影響とトレードオフ
- **代替案**: 検討したが選ばなかった選択肢
```

### B. 条件付きドキュメント（中優先度時）

#### 1. インラインコメント（選択的）
- 反直感的なコード：「なぜ」の説明のみ
- ビジネスルール：ドメイン知識の橋渡し
- 一時的な回避策：将来の改善点の明記

#### 2. README（プロジェクト概要）
- 5分で理解できる概要
- セットアップ手順（自動化できない部分のみ）
- 主要な概念とアーキテクチャ決定

### C. 避けるべきドキュメント（低優先度時）
- コードを読めば分かる「何を」の説明
- 自明な関数の詳細説明
- 頻繁に変更される実装詳細の文書化
- 形式的すぎる詳細仕様書

## 実装指針

### 1. 段階的アプローチ
1. **最小実装**: 最も重要な部分のみドキュメント化
2. **フィードバック収集**: チームの実際のニーズを把握
3. **漸進的拡張**: 価値が証明された部分を拡張
4. **継続的見直し**: 定期的に有効性を評価

### 2. 品質基準
- **正確性**: 実装との整合性
- **簡潔性**: 必要最小限の情報
- **実用性**: 実際の作業に役立つ内容
- **保守性**: 継続的更新が現実的

### 3. 自動化活用
- ドキュメント生成の自動化
- 実装とドキュメントの整合性チェック
- 陳腐化したドキュメントの検出

## 記述時の思考プロセス

### 質問フレームワーク
1. **必要性**: このドキュメントなしに他の開発者は理解できるか？
2. **持続性**: このドキュメントは長期間価値を保てるか？
3. **コスト**: 作成・保守コストは価値を上回るか？

### 記述原則
- **Why over What**: 理由と背景を重視
- **Future Self**: 6ヶ月後の自分が理解できる内容
- **Newcomer Perspective**: 説明の明瞭さと簡潔さを考慮
- **Iterative Refinement**: 完璧を求めず、漸進的改善

## 品質チェックリスト

### ドキュメント完全性チェック
- [ ] 役割と責務が明確に記述されている
- [ ] 関数の目的が具体的に説明されている
- [ ] 全てのパラメータに型と意味が記載されている
- [ ] 戻り値の型と意味が説明されている
- [ ] 発生する可能性のある例外が全て列挙されている
- [ ] 実用的な使用例が提供されている

### 記述品質チェック
- [ ] 技術的正確性：実装と完全に一致している
- [ ] 明確性：曖昧な表現や専門用語の説明不足がない
- [ ] 完全性：理解に必要な情報が全て含まれている
- [ ] 一貫性：プロジェクト内で統一された形式・用語を使用
- [ ] 実用性：実際の開発作業に役立つ内容である

### 保守性チェック
- [ ] 実装変更時に更新しやすい構造になっている
- [ ] 自動生成ツールとの互換性がある
- [ ] バージョン管理システムで追跡可能
- [ ] レビュープロセスに組み込まれている

## 実装手順

### 1. 新規作成時
1. 関数・クラス・構造体・プロトコルの定義直後にドキュメントコメントを追加
2. 必須項目を全て記入（空欄やTODOは残さない）
3. 言語に合わせて記載

### 2. 既存コード更新時
1. 実装変更前にドキュメントの現在の正確性を確認
2. 実装変更と同時にドキュメントを更新
3. 変更内容がドキュメントに正しく反映されているか検証

### 3. レビュープロセス
1. **実装レビュー**: コードとドキュメントの整合性確認
2. **ドキュメントレビュー**: 必須項目の完全性と品質確認
3. **使用例確認**: サンプルコードの動作検証
4. **一貫性チェック**: プロジェクト全体での統一性確認
