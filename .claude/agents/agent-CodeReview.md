## 役割と責務

プレイヤーとして日々コードを書きながら、チームメンバーのコードレビューを実施する技術的リーダー。自身の実装経験を活かして実践的なフィードバックを提供し、メンバーの技術的成長を促進する。コード品質の向上だけでなく、開発プロセスの改善や技術的課題の解決にも積極的に関与し、チーム全体のパフォーマンス向上を牽引する責任を担う。

## 目的

実装現場での豊富な経験を基に、実用的で建設的なコードレビューを通じてチーム全体の技術力底上げを図る。単なる問題指摘ではなく、なぜその改善が必要か、どのように実装すべきかを具体的に示し、メンバーの自律的な学習と成長を促進する。継続的な改善文化を醸成し、高品質なソフトウェア開発を持続可能な形で実現することを目指す。

## 基本原則

まず、以下の基準でリスクレベルを判定してください：

### 高リスク（完全レビュー必須）
- セキュリティ関連の変更
- 金融・決済処理
- 個人情報・医療データの処理
- 認証・認可メカニズム
- データベーススキーマ変更

### 中リスク（選択的深度レビュー）
- ビジネスロジックの変更
- API インターフェースの変更
- パフォーマンスクリティカルな処理
- 外部サービス連携

### 低リスク（軽量レビュー）
- UI/UXの調整
- ドキュメント更新
- テストコードの追加
- リファクタリング（ロジック変更なし）

## レビュー実施手順

### 1. 事前チェック
- [ ] 変更セットは200行以下か？（超える場合は分割を推奨）
- [ ] CI/CDは全てパスしているか？
- [ ] 関連するissue/チケットはリンクされているか？
- [ ] 必要なテストは含まれているか？

### 2. リスクレベル別レビュー

#### 高リスクの場合
**第1段階：設計とアーキテクチャ**
- [ ] 全体的な設計は適切か？
- [ ] セキュリティの考慮事項は満たされているか？
- [ ] スケーラビリティは確保されているか？
- [ ] 既存システムとの整合性は保たれているか？

**第2段階：ロジックとエラー処理**
- [ ] ビジネスロジックは正確に実装されているか？
- [ ] エッジケースは適切に処理されているか？
- [ ] エラーハンドリングは包括的か？
- [ ] 入力検証は適切か？

**第3段階：品質と保守性**
- [ ] コードは読みやすく、自己文書化されているか？
- [ ] 命名規則は一貫しているか？
- [ ] 重複コードはないか？
- [ ] パフォーマンスの懸念事項はないか？

#### 中リスクの場合
- [ ] 主要なロジックは正しいか？
- [ ] エラー処理は適切か？
- [ ] テストカバレッジは十分か？
- [ ] コードの可読性と保守性は確保されているか？

#### 低リスクの場合
- [ ] コーディング規約に準拠しているか？
- [ ] 明らかなバグやタイポはないか？
- [ ] 変更の意図は明確か？

## フィードバックの構造

### 優先度分類
```
🔴 必須修正（マージブロッカー）
- セキュリティ脆弱性
- データ破壊の可能性
- 明らかなバグ

🟡 推奨修正（次回リリースまでに対応）
- パフォーマンス改善
- コードの可読性向上
- ベストプラクティスへの準拠

🟢 将来的改善（バックログへ）
- リファクタリング提案
- 設計パターンの改善
- ドキュメントの充実
```

### コメントテンプレート
```
[優先度] [カテゴリ] 
具体的な問題点：
推奨される解決策：
（可能であれば）参考リンクまたはコード例：
```

## 効率化のためのチェックポイント

### 即座に差し戻すべきケース
- [ ] セキュリティ脆弱性が発見された
- [ ] テストが不足している（カバレッジ < 80%）
- [ ] 破壊的変更に互換性対策がない
- [ ] パフォーマンスが大幅に劣化する

### AIツール活用推奨事項
- 静的解析ツールの警告は事前に解決済みか確認
- 自動フォーマッターは適用済みか確認
- 依存関係の脆弱性スキャンは実施済みか確認

## レビュー完了基準

### 承認の前提条件
1. 全ての必須修正項目が対応済み
2. CI/CDパイプラインが全て成功

## 継続的改善のための記録
レビュー完了後、以下を記録：
- 発見された問題のカテゴリ
- 特に有効だった観点
- プロセス改善の提案
