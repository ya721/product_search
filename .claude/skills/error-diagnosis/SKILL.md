# Error Diagnosis Skill
 
エラーの原因究明から解決・再発防止までを体系的に支援する。
 
---
 
## 基本原則
 
- **証拠ファースト**：エラーメッセージ・ログ・実行結果など、実証的な情報を最優先にする
- **仮説を明示**：「〜の可能性があります」と「〜が原因です」を明確に区別する
- **最小確認単位**：1ステップごとに確認コマンドを提示し、結果を受け取ってから次へ進む
- **再発防止**：修正だけでなく、根本原因の解消と予防策まで提示する
---
 
## 診断フロー
 
### Step 1：症状の把握
 
ユーザーから以下を収集する（すでに提示されていれば収集不要）：
 
- エラーメッセージ・ログの全文
- 実行したコマンドまたは操作
- 期待した動作と実際の動作
- 環境情報（OS・ランタイム・バージョンなど）
情報が不足している場合は、最小限の質問を1つだけ行う。
 
---
 
### Step 2：原因の特定
 
#### 出力フォーマット
 
```
## エラー原因
 
**症状**
（何が起きているか）
 
**根本原因**
（なぜ起きているか）
 
**確認コマンド**（仮説がある場合）
（原因を確定するためのコマンド）
```
 
#### 診断の優先順位
 
1. エラーメッセージの最終行・exit code を最初に確認する
2. 環境固有の問題（PATH・権限・バージョン不一致）を疑う
3. タイミング・実行順序の問題を疑う（自動実行 vs 手動実行など）
4. 設定ファイルの内容と実際の動作の乖離を確認する
#### よくある原因パターン
 
| カテゴリ | 典型的な原因 |
|---|---|
| コマンド not found | PATH未設定・インストール先の不一致 |
| 権限エラー | sudo不在・root/非rootの混在 |
| 設定反映漏れ | 古い設定が残存・スキップ条件に合致 |
| ネットワーク | DNS解決失敗・プロキシ・ドメイン制限 |
| タイミング | 認証前実行・依存サービス未起動 |
| 環境差異 | ローカル vs CI vs コンテナで異なる環境変数 |
 
---
 
### Step 3：修正案の提示
 
```
## 修正方法
 
**即時対処**（今すぐ動かす）
（コマンドまたは設定変更）
 
**恒久対処**（再発防止）
（スクリプト修正・設定ファイル変更など）
 
**確認**
（修正後に実行すべき検証コマンド）
```
 
複数の修正案がある場合は比較表で提示し、推奨案を明示する。
 
---
 
### Step 4：検証
 
ユーザーから実行結果を受け取り、以下を判断する：
 
- **解決済み**：根本原因の解消と再発防止策を確認して完了
- **新たな問題が発生**：Step 2 に戻り、新しいエラーを診断する
- **改善なし**：仮説を見直し、別の原因を検討する
---
 
### Step 5：再発防止（解決後）
 
```
## 再発防止
 
**根本原因の要約**
（何が問題だったか）
 
**今回の修正内容**
（何を変えたか）
 
**今後の注意点**
（同種の問題を防ぐために）
```
 
---
 
## 出力スタイル
 
- コードブロックには必ず言語を明示する（`bash`・`json`・`python`など）
- 確認コマンドと修正コマンドは必ず分けて提示する
- 「おそらく」「可能性がある」など不確実な表現は根拠とセットで使用する
- 解決の見込みが低い方向には深追いせず、別の仮説に切り替える
