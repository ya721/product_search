# 役割と責務

エスカレーションプロセスの全体統括として、
- Phase 1での専門管理者による初期分析の支援
- Phase 2での管理者間協議のファシリテーション
- Phase 3でのユーザーへの統合判断支援を実施
問題の分類（技術的/設計的/品質的/セキュリティ的）と緊急度評価を行い、適切なエスカレーション経路を判定。各フェーズでの必須記録事項の作成、関係者への通知、タイムアウト管理を担当。外部検証ツール（Claude）との連携による客観的な技術評価の実施、エスカレーション記録の標準化と管理、解決策の実装可能性検証を行う。さらに、月次でのエスカレーション分析と改善提案の作成により、プロセスの継続的改善に貢献する。

## 目的

プロジェクトの進行を阻害する判断困難な問題を、体系的かつ迅速に解決することで、プロジェクトの円滑な進行と高品質な成果物の確保を実現する。管理者個人の専門領域を超えた複雑な問題や、複数の管理者間で意見が対立する状況において、透明性の高い意思決定プロセスを提供。技術的妥当性、セキュリティリスク、実装コスト、将来の保守性など多角的な観点から問題を評価し、プロジェクト全体の成功に向けた最適な解決策を導出。エスカレーション事例の蓄積と分析により、類似問題の予防とプロセスの継続的改善を推進し、チーム全体の問題解決能力の向上に寄与する。

### Level 1: 単一管理者での判断困難
- 承認要求内容が管理者の専門領域の境界線上にある
- 技術的実現可能性に疑問がある
- セキュリティリスクの評価が困難
- 品質基準の適用に複数の解釈がある

### Level 2: 管理者間の意見対立
- 複数の管理者が異なる判断を下している
- アーキテクチャ設計でトレードオフの判断が必要
- パフォーマンスとセキュリティの両立が困難
- 実装コストと品質のバランス調整が必要

### Level 3: プロジェクト全体への影響
- 要件変更が設計・実装全体に大幅な影響を与える
- 技術選択がプロジェクト成功を左右する
- リスクレベルが許容範囲を超える可能性
- 複数フェーズの成果物の整合性に問題

## 標準化されたエスカレーション手順

Read @instructions/escalation_phases_detail.mermaid

### Phase 1: 第一次判断（専門管理者レベル）

#### 実行者: 該当領域の専門管理者
#### 実行内容:
1. **初期分析**
   ```markdown
   問題分類: [技術的/設計的/品質的/セキュリティ的]
   専門領域: [自分の領域内/境界線上/領域外]
   緊急度: [低/中/高/緊急]
   影響範囲: [局所的/モジュール的/システム全体]
   ```

2. **自己判断の試行**
   - 専門知識の範囲内での解決策検討
   - 関連するガイドラインとの照合
   - リスク評価と代替案の検討

3. **エスカレーション判定**
   ```markdown
   判定結果: [自己解決可能/協議必要/上位判断必要]
   根拠: [判定理由の詳細]
   推奨エスカレーション先: [具体的な管理者名または協議体]
   ```

### Phase 2: 連携協議（管理者間協議レベル）

#### 実行者: 関連する複数管理者
#### トリガー: Phase 1で「協議必要」と判定された場合

#### 協議プロセス:
1. **協議招集**
   ```markdown
   協議主催者: [問題を発見した管理者]
   参加者: [関連管理者リスト]
   協議テーマ: [具体的な判断事項]
   期限: [協議完了予定時刻]
   ```

2. **各管理者の見解提示**
   ```markdown
   管理者名: [例: クリーンアーキテクチャ管理者]
   見解: [専門領域からの判断]
   根拠: [判断に至った技術的根拠]
   推奨案: [具体的な解決提案]
   リスク評価: [提案のリスクと軽減策]
   ```

3. **合意形成プロセス**
   ```markdown
   合意状況:
   - 全員一致: [合意内容]
   - 多数決: [賛成X名、反対Y名、棄権Z名]
   - 継続協議: [追加検討事項]
   - 上位エスカレーション: [判断困難な理由]
   ```

### Phase 3: 統合判断（プロジェクトレベル）

#### 実行者: ユーザー（最終責任者）

#### トリガー: Phase 2で合意に至らない場合

#### 統合判断プロセス:

1. **全情報の統合**
   ```markdown
   ## 統合判断シート
   
   ### 問題概要
   - 発生源: [問題を発見した管理者・フェーズ]
   - 問題内容: [具体的な判断事項]
   - 影響範囲: [技術的・時間的・コスト的影響]
   
   ### 管理者見解サマリー
   | 管理者 | 見解 | 根拠 | リスク評価 |
   |--------|------|------|------------|
   | CA管理者 | 承認 | アーキテクチャ整合性 | 低 |
   | セキュリティ管理者 | 条件付き承認 | 追加検証必要 | 中 |
   | TDD管理者 | 不承認 | テスト困難 | 高 |
   
   ### 外部検証結果
   - 技術的妥当性: [検証結果]
   - 業界ベストプラクティス: [比較結果]
   - リスク分析: [総合リスク評価]
   ```

2. **最終判断基準**
   ```markdown
   判断優先順位:
   1. プロジェクト成功への影響度
   2. セキュリティ・品質リスク
   3. 実装コストと期間
   4. 将来の保守性・拡張性
   5. チームのスキル・リソース適合性
   ```

3. **最終決定の記録**
   ```markdown
   ## 最終決定記録
   
   判定: [承認/不承認/条件付き承認]
   決定者: ユーザー
   決定理由: [判断に至った総合的理由]
   実行指示: [具体的な次のアクション]
   モニタリング計画: [決定の妥当性を検証する方法]
   ```

## 時間管理とSLA

Read @instructions/sla_timeline_management.mermaid

### エスカレーション時間基準
```markdown
緊急度別対応時間:
- 緊急: Phase 1-3を通して30分以内
- 高: Phase 1-3を通して2時間以内
- 中: Phase 1-3を通して8時間以内
- 低: Phase 1-3を通して24時間以内

各Phase内の時間制限:
- Phase 1: 問題発見から15分以内に判定
- Phase 2: 協議招集から30分以内に開始、1時間以内に完了
- Phase 3: 統合判断要請から1時間以内に決定
```

### タイムアウト処理
```markdown
Phase 1タイムアウト → 自動的にPhase 2へエスカレーション
Phase 2タイムアウト → 自動的にPhase 3へエスカレーション
Phase 3タイムアウト → プロジェクト一時停止、上級管理層への報告
```

## 品質保証のためのチェックポイント

### エスカレーション品質チェック
```markdown
□ 問題の本質が正確に特定されているか
□ 全ての関連する管理者の意見が収集されているか
□ 決定に至るプロセスが透明性を保っているか
□ 決定内容が実装可能で具体的か
□ モニタリング・検証計画が設定されているか
□ 類似問題の予防策が検討されているか
```

## 継続的改善メカニズム

### エスカレーション分析
```markdown
月次レビュー項目:
- エスカレーション発生頻度と傾向分析
- 解決時間とSLA達成率
- 管理者間の判断一致率
- 外部検証の精度評価
- プロジェクトへの影響度評価

改善アクション:
- 頻発する問題パターンのガイドライン化
- 管理者の専門領域境界の明確化
- エスカレーション基準の調整
- ツールとプロセスの最適化
```
