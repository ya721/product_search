# レビュー方針（リスクベース）

あなたの責務は、**変更内容のリスクレベルを評価**し、**そのレベルに応じた適切な深さでコードレビュー**を実施することです。

***

## 0. リスクレベル判定（最初に実施）

以下の基準でリスクレベルを判定する。**複数に該当する場合は最も高いリスクレベルを採用**する。

### 高リスク（完全レビュー必須）

*   セキュリティ関連の変更（例：暗号化方式、入力検証、XSS/SQLi対策）
*   金融・決済処理（課金、返金、会計連携）
*   個人情報・医療データの処理（PII/PHI）
*   認証・認可メカニズム（セッション、トークン、RBAC/ABAC）
*   データベーススキーマ変更（移行、破壊的変更）

**境界例**：UI 変更だが認証チェックを追加 → **高リスク扱い**

### 中リスク（選択的深度レビュー）

*   ビジネスロジックの変更（料金計算、在庫、締め処理）
*   API インターフェースの変更（契約/スキーマ、互換性）
*   パフォーマンスクリティカルな処理（バッチ、クエリ最適化）
*   外部サービス連携（新規/主要バージョンアップ）

**境界例**：API レスポンスに非互換なフィールド追加 → **中〜高リスクの間だが、互換性破壊なら高**

### 低リスク（軽量レビュー）

*   UI/UX の調整（文言、スタイル、軽微な表示ロジック）
*   ドキュメント更新
*   テストコードの追加（ロジック変更なし）
*   リファクタリング（挙動不変）

***

## 1. 事前チェック（共通）

*   [ ] 変更セットは **200行以下** か？（超える場合は**分割**を推奨）
*   [ ] **CI/CD は全てパス** しているか？
*   [ ] 関連する **issue/チケットがリンク** されているか？
*   [ ] **必要なテスト**（ユニット/統合/回帰）が含まれているか？
*   [ ] 静的解析・自動フォーマッターは **適用済み** か？
*   [ ] 依存関係の **脆弱性スキャン済み** か？

***

## 2. リスクレベル別レビュー手順

### 高リスク（3段階で実施）

**第1段階：設計とアーキテクチャ**

*   [ ] 全体設計は目的に対して妥当か？
*   [ ] セキュリティ要件（最小権限、入力検証、秘密情報の取り扱い）は満たされているか？
*   [ ] スケーラビリティ・可用性の配慮はあるか？
*   [ ] 既存システムとの整合性・互換性は保たれているか？

**第2段階：ロジックとエラー処理**

*   [ ] ビジネスルールは仕様通りか？
*   [ ] エッジケース（境界値、空/null、タイムゾーン、再送/重複処理）が考慮されているか？
*   [ ] エラーハンドリング（リトライ、タイムアウト、フォールバック、監視/通知）は包括的か？
*   [ ] 入力/出力の **バリデーション** は適切か？

**第3段階：品質と保守性**

*   [ ] コードの可読性（命名、一貫性、凝集度、責務分離）は高いか？
*   [ ] 重複・不要コード、過剰最適化がないか？
*   [ ] パフォーマンス懸念（N+1、不要な同期 I/O、大きなオブジェクトコピー等）はないか？
*   [ ] ログに機密情報が含まれないか？（マスキング・レベル適正）

### 中リスク（主要観点＋必要に応じて品質項目を参照）

*   [ ] 主要ロジックは正しく、仕様に整合しているか？
*   [ ] エラー処理は適切か？（異常系テスト含む）
*   [ ] テストカバレッジは十分か？（重要経路は網羅）
*   [ ] 可読性・保守性に大きな懸念はないか？
*   （必要に応じて **高リスクの「品質と保守性」チェック** を参照）

### 低リスク（軽量確認）

*   [ ] コーディング規約・フォーマットに準拠しているか？
*   [ ] 明らかなバグ、タイポ、不要差分はないか？
*   [ ] 変更の意図・範囲が明確か？（説明/コメント/コミットメッセージ）

***

## 3. フィードバックの構造

### 優先度分類

    🔴 必須修正（マージブロッカー）
    - セキュリティ脆弱性
    - データ破壊の可能性
    - 明らかなバグ

    🟡 推奨修正（次回リリースまでに対応）
    - パフォーマンス改善
    - コードの可読性向上
    - ベストプラクティスへの準拠

    🟢 将来的改善（バックログへ）
    - リファクタリング提案
    - 設計パターンの改善
    - ドキュメントの充実

***

## 4. 即座に差し戻すべきケース（例示）

*   [ ] セキュリティ脆弱性が発見された（例：未サニタイズ入力、秘匿情報のログ出力）
*   [ ] テストが不足している（重要経路の欠落、**カバレッジ < 80%**）
*   [ ] 破壊的変更に互換性対策（移行/段階的リリース）がない
*   [ ] パフォーマンスが大幅に劣化する（実測/推定インパクトが大）
*   [ ] 本番環境に影響する危険な環境依存コード/設定が含まれる
*   [ ] ログ/例外メッセージに機密情報（PII、トークン、鍵）が含まれる恐れ

***

## 5. AI/自動化ツールの活用（推奨）

*   静的解析（Lint/型/セキュリティ）の警告は **事前解消** 済みか確認
*   自動フォーマッター（例：Prettier、Black、gofmt）を **適用**
*   依存関係の **脆弱性スキャン**（例：Dependabot、Snyk）結果を確認
*   変更差分に対する **AIレビューの一次チェック** を活用（人手レビューの補助に限定）

***

## 6. レビュー完了基準（承認前提）

1.  **全ての必須修正（🔴）が解消** されている
2.  **CI/CD パイプラインが全て成功** している
3.  （高/中リスクの場合）**回帰の主要テストが合格** している
4.  ドキュメント更新（仕様/API/運用手順）が必要な場合は反映済み

***

## 7. 継続的改善のための記録（レビュー後）

*   発見された問題の **カテゴリ**（セキュリティ/性能/設計/可読性/テスト 等）
*   特に有効だった **レビュー観点**（再発防止に役立つ示唆）
*   **プロセス改善** の提案（テンプレート/自動化/定義の見直し）
*   **対応工数の目安**（レビュー時間、修正サイクル回数）

***

### 付記：運用上のメモ

*   **リスクが重なった場合は高い方を採用**（原則）
*   不明点や仕様の曖昧さがある場合は、**早期に質問** → **合意形成後にレビュー継続**
*   大規模差分（>200行）は、**スコープを分割** して段階的にレビュー
