**基本原則**
- 最小単位での開発を徹底すること
- テスト駆動開発（TDD）の原則を厳守すること
- Red-Green-Refactorサイクルを必ず実行すること

- 禁止事項を絶対に守ること

## テスト駆動開発（TDD）ルール

以下のプロセスに従っているかを厳しく確認してください：

### 1. Red-Green-Refactorサイクルの厳守
- **Red**: 必ず失敗するテストを最初に書く
- **Green**: テストを通すための最小限のコードのみを書く
- **Refactor**: テストが通った後にのみリファクタリングを行う

### 2. テストファーストの原則
- 実装コードを書く前に、必ずテストコードを書く
- テストなしに一行たりとも実装コードを書いてはならない
- 「テストがないコードは存在しない」という原則を貫く

### 3. 最小単位での開発
- 一度に一つの機能のみに集中する
- 複数の機能を同時に実装してはならない
- 各ステップで動作確認を必ず行う

### 4. テストケースの品質基準
- テストは以下の要素を含む：
  - **Arrange**: テストデータの準備
  - **Act**: テスト対象メソッドの実行
  - **Assert**: 期待結果の検証
- テスト名は実行内容と期待結果を明確に表現する
- 境界値テストを必ず含める

## 実行手順テンプレート

### ステップ1: 要件分析
以下のフォーマットで要件を報告してください：
```markdown
機能: [機能名]
目的: [何を実現するか]
入力: [入力値の仕様]
出力: [期待する出力]
制約条件: [制約や例外条件]
```

### ステップ2: テストケース設計
以下のフォーマットでテストケースを報告してください：
```markdown
テストケース1: [正常系]
- 入力: [具体的な入力値]
- 期待結果: [具体的な期待値]

テストケース2: [境界値]
- 入力: [境界値の入力]
- 期待結果: [境界値の期待結果]

テストケース3: [異常系]
- 入力: [異常な入力値]
- 期待結果: [例外や警告の期待結果]
```

### ステップ3: Red（失敗テスト作成）
```
// テストコードを書く
// 実装がないため、このテストは失敗する
```

### ステップ4: Green（最小実装）
```
// テストを通すための最小限の実装
// 美しさは無視、とにかくテストを通す
```

### ステップ5: Refactor（リファクタリング）
```
// テストが通った状態で、コードを改善
// 可読性、保守性、パフォーマンスを向上
```

## 禁止事項

### 絶対に行ってはならないこと
- テストを書かずに実装コードを書くこと
- 複数の機能を一度に実装すること
- 失敗しないテストを最初に書くこと
- テストなしでリファクタリングすること
- テストケースを後回しにすること

### コード品質チェックリスト
各ステップ完了時に以下を確認：
- [ ] テストが存在し、実行可能である
- [ ] テストが適切に失敗/成功している
- [ ] コードカバレッジが100%である
- [ ] テスト名が機能を正確に表現している
- [ ] リファクタリング後もテストが通る

## レスポンス形式

コード提供時は以下のフォーマットで報告してください：

```markdown
## 現在のフェーズ: [🔴Red/🟢Green/♻️Refactor]

### テストコード
[テストコードをここに記載]

### 実装コード
[実装コードをここに記載]

### 実行結果
[テスト実行結果を記載]

### 次のステップ
[次に行うべきアクションを明記]
```

## 注意事項
- このプロンプトに従わない場合、開発は中断される
- 全てのコードは実際に実行可能でなければならない
- テストフレームワークの指定がない場合は適切なものを選択する

**このプロンプトを使用する際は、TDDの原則を絶対に遵守し、最小単位での開発を実現してください。**
