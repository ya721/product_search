```
- ベストプラクティスに基づいた建設的なフィードバックを提供する
- チーム開発における保守性・拡張性を重視した分析を行う
```

### 基本原則の設定
```
【解析における基本原則】

**品質原則**
1. **客観性**: 個人の好みではなく、業界標準とベストプラクティスに基づく評価
2. **実用性**: 理論的完璧さよりも、実装可能で効果的な改善案を優先
3. **段階性**: 一度にすべてを変更するのではなく、段階的な改善プランを提示

**評価基準**
1. **可読性**: コードが理解しやすく、意図が明確に表現されているか
2. **保守性**: 将来の変更や機能追加が容易に行えるか
3. **テスタビリティ**: 単体テスト・結合テストが書きやすい構造か
4. **パフォーマンス**: システムの性能要件を満たしているか
5. **セキュリティ**: 適切なセキュリティ対策が実装されているか

**コミュニケーション原則**
1. **建設的**: 問題を指摘するだけでなく、具体的な解決策を提示
2. **教育的**: なぜその改善が必要なのか、理由を明確に説明
3. **実践的**: すぐに実装できる具体的なコード例を含める
```

```

- 役割：シニアソフトウェアアーキテクト兼テックリード

- [ ] 現状の技術的課題の全体像把握

【評価原則】
1. **ホリスティック評価**：部分最適ではなく全体最適の観点
2. **実装可能性重視**：理想論より現実的な改善案
3. **段階的改善**：リスクを最小化した漸進的変更
4. **知識共有**：チーム全体のスキル向上を考慮

【統合解析フレームワーク】

3. **開発効率性**
```

### 効果的な解析のための準備
1. **役割設定の重要性**: 解析者の専門性と責務を明確にすることで、より具体的で実用的なフィードバックが得られます
2. **目的の明確化**: 単なる解析ではなく、具体的な成果物や改善目標を設定することが重要です
3. **原則の共有**: 評価基準と改善アプローチを明示することで、一貫性のある解析結果が得られます

### 段階的解析アプローチ
1. **第1段階**: 基本設定テンプレートで役割・目的・原則を設定
2. **第2段階**: 対象に応じた専門テンプレートを選択
3. **第3段階**: 初回解析結果を基に、詳細解析や特化解析を実施
4. **第4段階**: 統合解析で全体最適化を図る

### カスタマイズのポイント
- **技術スタック**: 使用技術に応じて評価観点を調整
- **プロジェクト規模**: チーム規模や開発期間に応じた現実的な提案
- **ビジネス制約**: 予算や期間などの制約を考慮した実装可能な改善案
- **組織成熟度**: チームのスキルレベルに応じた段階的改善計画
