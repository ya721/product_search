## 役割と責務

開発プロセス全段階でのセキュリティ実装を主導し、設計段階での脅威特定から実装時の具体的対策指導、コードレビューでの厳格なチェックリスト適用まで実践的に担当。AI/MLセキュリティやサプライチェーンセキュリティなど新興脅威への対応策を策定・実行。ツール選択・導入から偽陽性対策まで技術的な問題解決を行いつつ、チームメンバーへの教育・指導を通じてセキュリティ文化を醸成します。

## 目的

開発速度を損なうことなく、実用的で効果的なセキュリティ対策を組織に根付かせること。単発的なセキュリティチェックではなく、継続的改善プロセスとして機能する仕組みを構築。多層防御の考え方で複数の防御層を実装し、組織のリスク許容度とリソースに見合った現実的なセキュリティレベルを達成。最終的には、セキュリティが開発チームの自然な習慣として定着し、持続可能なセキュアな開発環境を実現します。

## 基本原則

- **実用的セキュリティ**: 理論的完璧性よりも実装可能で効果的な対策を優先します
- **継続的改善**: セキュリティは一度の実装ではなく、継続的な改善プロセスです
- **リスクベースアプローチ**: 組織のリスク許容度とリソースに応じた現実的な対策を選択します
- **多層防御 (Defense in Depth)**: 単一のセキュリティ対策に依存せず、複数の防御層を構築します

## 開発ワークフローへの統合

このガイドラインは、`Programmer-Workflow-instructions.md` の各プロセスと密接に連携します。

- **設計段階 (プロセス 4)**: プログラマーは、実装する機能に関連する脅威を「主要脅威と対策」セクションから特定し、設計にセキュリティ対策を組み込んでください。
- **実装段階 (プロセス 5)**: プログラマーは、設計に基づき、このガイドラインに記載された具体的な対策をコードに実装してください。
- **コードレビュー段階 (プロセス 6)**: レビュアーは、「レビューチェックリスト（優先度付き）」を**必須の確認項目**として使用し、セキュリティ要件が満たされているかを厳格に検証してください。
- **承認プロセス**:
  - レビュアーは、チェックリストの項目が全て満たされていることを確認した場合にのみ、レビューを承認できます。
  - 🔴 **Critical** な問題が発見された場合、レビュアーは即座にプルリクエストを差し戻し、プログラマーに修正を要求してください。
  - 判断に迷うセキュリティ上の懸念がある場合は、必ずセキュリティ管理者にエスカレーションし、承認を得てください。

## 組織成熟度別アプローチ

### スタートアップ・小規模組織
- **重点**: 基本的なセキュリティ衛生管理
- **推奨ツール**: 無料・オープンソースツールの活用
- **実装スピード**: 迅速な実装と反復改善

### 中規模組織
- **重点**: 自動化とプロセス標準化
- **推奨ツール**: 商用ツールとオープンソースの併用
- **実装スピード**: 段階的導入と効果測定

### 大規模・エンタープライズ
- **重点**: 包括的ガバナンスとコンプライアンス
- **推奨ツール**: エンタープライズ級ソリューション
- **実装スピード**: 慎重な計画と大規模展開

## 主要脅威と対策

### 1. アクセス制御の不備（継続的最重要脅威）
**現状**: `claude -p "アクセス制御の不備の最新調査"`
**指示**:
- [ ] **ゼロトラスト原則**: 全てのリクエストを検証・認証
- [ ] **API レベル認可**: REST/GraphQL APIで細粒度の認可制御
- [ ] **動的権限評価**: ユーザーコンテキストに応じたリアルタイム権限チェック
- [ ] **セッション管理**: JWT トークンの適切な実装と有効期限管理

### 2. ソフトウェアサプライチェーンセキュリティ（急速拡大する脅威）
**現状**: `claude -p "ソフトウェアサプライチェーンセキュリティの最新調査"`
**指示**:
- [ ] **Software Bill of Materials (SBOM)**: 全依存関係の可視化と管理
- [ ] **多段階検証**: 自動スキャン + 手動レビュー + 専門家評価
- [ ] **サプライヤー評価**: 依存関係提供者のセキュリティ成熟度評価
- [ ] **インシデント対応計画**: サプライチェーン攻撃時の迅速な対応手順

**ツール選択ガイド**:
```
段階1（基本）: OWASP Dependency-Check（偽陽性対策付き）
段階2（改善）: Snyk, WhiteSource, Veracode等の商用SCA
段階3（高度）: カスタム分析ツール + 脅威インテリジェンス統合
```

### 3. 暗号化の失敗（継続的重要脅威）
**現状**: `claude -p "暗号化の失敗に関する最新調査"`
**指示**:
- [ ] **現代的暗号化**: TLS 1.3、AES-256-GCM、ChaCha20-Poly1305の採用
- [ ] **鍵管理の自動化**: AWS KMS, Azure Key Vault, HashiCorp Vaultの統合
- [ ] **証明書管理**: 自動更新とローテーション（Let's Encrypt等）
- [ ] **暗号化監査**: 定期的な暗号実装レビューと脆弱性評価

### 4. インジェクション攻撃（進化する脅威）
**現状**: `claude -p "インジェクション攻撃に関する最新調査"`
**指示**:
- [ ] **NoSQL/Graph インジェクション**: MongoDB, Neo4j等への対策
- [ ] **API インジェクション**: GraphQL, REST APIの入力検証強化
- [ ] **コンテンツセキュリティポリシー (CSP)**: XSS攻撃の多層防御
- [ ] **入力検証ライブラリ**: 言語・フレームワーク固有の検証ツール統合

### 5. AI/ML セキュリティ（新興脅威）
**現状**: `claude -p "AI/ML セキュリティに関する最新調査"`
**指示**:
- [ ] **AI生成コードレビュー**: 人間による必須レビュープロセス実装
- [ ] **プロンプトインジェクション対策**: LLM統合アプリケーションの入力検証
- [ ] **モデル完全性**: 使用するAIモデルの検証と監査
- [ ] **データプライバシー**: AI学習データでの機密情報漏洩防止

## ツール選択マトリックス

### SCAツール（Software Composition Analysis）

| 組織規模 | 推奨ツール | 理由 | 偽陽性対策 |
|---------|-----------|------|------------|
| 小規模 | OWASP Dependency-Check | 無料、CI/CD統合容易 | 抑制ファイル活用、手動レビュー |
| 中規模 | Snyk, Mend (旧WhiteSource) | 精度向上、自動修正提案 | インテリジェントフィルタリング |
| 大規模 | Veracode, Black Duck | エンタープライズ機能、ガバナンス | カスタマイズ可能なポリシー |

### SAST/DASTツール

| カテゴリ | オープンソース | 商用 | 推奨用途 |
|---------|----------------|------|----------|
| SAST | SonarQube, Semgrep | Checkmarx, Veracode | コード品質+セキュリティ |
| DAST | OWASP ZAP | Burp Suite Professional | 実行時脆弱性検出 |
| IAST | - | Contrast Security, Seeker | 運用環境での継続監視 |

## レビューチェックリスト（優先度付き） - コードレビュー時に厳格に適用

レビュアーは、以下のチェックリストを**厳格に**適用し、プルリクエストを評価してください。🔴 Critical な項目が一つでも存在する場合、そのプルリクエストは即座に差し戻されます。

### 🔴 Critical（ビルド失敗級）
- [ ] 機密情報のハードコーディング（API keys, passwords, tokens）
- [ ] SQLインジェクション脆弱性（パラメータ化クエリ未使用）
- [ ] 認証・認可の完全なバイパス
- [ ] 既知の重大脆弱性（CVSS 9.0+）を持つ依存関係

### 🟡 High（必須修正）
- [ ] XSS脆弱性（入力検証・出力エスケープ不備）
- [ ] 不適切なエラーハンドリング（機密情報漏洩）
- [ ] 弱い暗号化アルゴリズム（MD5, SHA-1, DES等）
- [ ] CVSS 7.0-8.9の脆弱性を持つ依存関係

### 🟢 Medium（推奨修正）
- [ ] セキュリティヘッダーの不備（CSP, HSTS等）
- [ ] ログ不備（認証失敗、特権操作の未記録）
- [ ] 入力値検証の不完全性
- [ ] CVSS 4.0-6.9の脆弱性を持つ依存関係
